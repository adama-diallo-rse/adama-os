Cadre posé, matérialité tranchée, données structurées : tout est prêt pour la dernière marche. Assembler un pipeline qui transforme la donnée brute en rapport ESRS, et qui se rejoue quand les données changent. Cette leçon décrit l'architecture, étape par étape.

## Voir le rapport comme un flux

Un rapport automatisé n'est pas un document que l'on tape, c'est la **sortie d'un flux**. Ce flux enchaîne quatre étapes stables :

1. **Ingestion** : les données entrent depuis leurs sources aux points de collecte définis.
2. **Transformation** : les valeurs brutes deviennent des datapoints (calculs, facteurs, agrégations).
3. **Assemblage** : les datapoints se rangent dans la structure ESRS retenue par la matérialité.
4. **Rendu** : le rapport se génère dans le format voulu (document, page web, export).

Chaque étape est isolée et testable. Changer une source ne casse pas le rendu ; changer un facteur d'émission ne demande pas de retoucher le texte.

## Ingestion : une entrée par source

Reprenez la cartographie de la leçon précédente. Pour chaque source :

- Les systèmes structurés (compta, paie, ERP) se connectent par **lecture directe**.
- Les relevés terrain passent par un **import cadré** (gabarit fixe, validation à l'entrée).
- Les données tierces (Scope 3) arrivent par **collecte auprès des fournisseurs**, avec traçabilité de l'origine.

L'important est que l'ingestion soit **idempotente** : relancer l'import ne duplique rien. Adama OS applique ce principe partout, y compris sur la veille réglementaire, où chaque élément porte un identifiant de nouveauté.

## Transformation : des calculs purs et documentés

C'est le cerveau du pipeline. Les règles à suivre :

- **Fonctions pures** : une même entrée donne toujours la même sortie, sans effet de bord. Le calcul carbone d'Adama OS repose sur des fonctions de ce type, avec des facteurs ADEME surchargeables.
- **Facteurs externalisés** : les coefficients (facteurs d'émission, taux) sont des données, pas du code figé. On les met à jour sans réécrire la logique.
- **Chaque datapoint garde sa méthode** : le lien entre la valeur brute et la valeur publiée reste explicite.

## Assemblage : la matérialité comme plan

Le rapport ne contient que les standards ressortis matériels. L'assemblage suit ce plan :

- ESRS 2 (informations générales) toujours présent.
- Les standards thématiques retenus, chacun avec ses datapoints.
- Les enjeux écartés documentés brièvement, avec leur justification.

Comme la matérialité est elle-même structurée, ce plan se génère à partir d'elle. Modifier l'analyse met à jour le sommaire du rapport automatiquement.

## Rendu et régénération

Le rendu est la partie la plus visible mais la moins critique : une fois les datapoints assemblés, produire un document ou une page est mécanique. Le vrai gain est la **régénération**. Quand une donnée change :

- on met à jour la source,
- le pipeline recalcule les datapoints concernés,
- le rapport se régénère, piste d'audit incluse.

Ce qui prenait des semaines chaque année devient une mise à jour incrémentale.

## Le rôle de STRATA et d'Adama OS

STRATA est l'outil qui industrialise ce pipeline pour les PME : collecte, calcul, assemblage ESRS, rendu. Adama OS en est la vitrine technique et le laboratoire : moteur de calcul carbone et de matérialité, simulateur VSME, veille réglementaire automatisée. Le niveau que vous venez de suivre décrit précisément la méthode que ces outils appliquent.

## À retenir

- Le rapport est la sortie d'un flux en quatre étapes : ingestion, transformation, assemblage, rendu.
- Ingestion idempotente, transformations pures, facteurs externalisés : la fiabilité vient de l'architecture.
- Le vrai gain n'est pas de produire le rapport une fois, mais de le régénérer sans effort.

Vous êtes maintenant autonome sur le reporting CSRD automatisé. Le niveau 2 applique la même rigueur à un cadre plus léger : le standard VSME pour les PME.
